# AI文章の品質チェックとリライトガイド

記事を生成した後は、このファイルの9項目を順番にチェックしてリライトしてください。
各項目に「修正プロンプト」があります。必要に応じてそのまま使ってください。

---

## 1. 語尾の単調さ対策

**問題**
「〜です。〜です。〜です。」と同じ語尾が連続し、文章が単調に聞こえる。

**AIに起きやすい理由**
生成モデルは直前の文脈から次のトークンを予測するため、同じパターンが繰り返されやすい。

**修正ルール**
- 同じ語尾は2回まで。3回目は別の語尾に変える
- 「になります」「となります」の多用を避ける
- 体言止め・疑問形・「〜でしょう」なども織り交ぜる

**修正プロンプト**
```
以下の文章を確認してください。同じ語尾が3回以上連続している箇所を探し、
語尾にバリエーションをつけてリライトしてください。
「です」「ます」「になります」「となります」の繰り返しに特に注意してください。

```

## 2. 感情不足対策

**問題**
情報の羅列になっていて、書き手の感想・気づき・体験の熱量が伝わらない。

**AIに起きやすい理由**
事実や手順を正確に伝えることを優先するため、感情的なニュアンスが抜け落ちやすい。

**修正ルール**
- 「実際にやってみて気づいたこと」「最初は戸惑ったが〜」などの一人称の感触を加える
- 感想は誇張せず、落ち着いたトーンで添える
- 全文に感情を入れる必要はない。1〜2箇所あれば十分

**修正プロンプト**
```
以下の文章は情報の羅列になっています。
執筆者が実際に体験したような一人称の感触（気づき・戸惑い・納得感など）を
自然な形で1〜2箇所加えてリライトしてください。
誇張や作り話は不要です。

[文章をここに貼る]
```

## 3. 読者が見えない文章の修正

**問題**
誰に向けて書いているか不明確で、読んでも「自分ごと」として受け取りにくい。

**AIに起きやすい理由**
汎用的な説明を生成しようとするため、読者を特定しない抽象的な文章になりやすい。

**修正ルール**
- 「Ubuntu を使い始めた方」「ターミナルが初めての方」など、状況を絞って語りかける
- 「あなた」は使わず、「〜の場合は」「〜の方は」で絞る
- 冒頭か導入部で「誰向けか」を明示する

**修正プロンプト**
```
以下の文章は読者が見えにくい抽象的な内容になっています。
Ubuntu初心者・AI作業環境を整えたい人を想定読者として、
「自分ごと」として読めるように語りかけのトーンでリライトしてください。

[文章をここに貼る]
```

## 4. 教科書っぽさの回避

**問題**
手順・説明が整理されすぎていて、マニュアルや教科書のような無機質さが出る。

**AIに起きやすい理由**
論理的な構造を重視するため、書き手の個性がそぎ落とされた形式的な文体になりやすい。

**修正ルール**
- 「〜することが推奨されます」「〜するのが一般的です」などの客観的表現を減らす
- 「私はこうしています」「こちらの方法が個人的には使いやすいです」など、主観を混ぜる
- 形式ばった接続詞（「また」「さらに」「なお」の連続）を整理する

**修正プロンプト**
```
以下の文章が教科書・マニュアルのような硬い印象になっています。
執筆者の個人的な視点や主観を自然に加えて、
「人が書いた文章」らしいトーンにリライトしてください。

[文章をここに貼る]
```

## 5. 抽象表現の具体化

**問題**
「簡単です」「便利です」「効率が上がります」など、具体性のない形容詞で終わっている。

**AIに起きやすい理由**
評価・感想を一言で済ませようとするため、根拠のない抽象表現が増えやすい。

**修正ルール**
- 「簡単」→ どう簡単なのか（ステップ数・所要時間・前提知識など）
- 「便利」→ 何が、どのように便利なのか
- 「効率が上がる」→ 何の作業が、どう変わるのか

**修正プロンプト**
```
以下の文章に「簡単」「便利」「効率的」「優れている」などの
根拠のない抽象的な表現が含まれています。
それぞれを具体的な内容（ステップ数・作業の変化・比較など）に置き換えてリライトしてください。

[文章をここに貼る]
```

## 6. 主張の弱さの改善

**問題**
「〜かもしれません」「〜とも言えます」「〜ではないでしょうか」が多く、意見がぼやける。

**AIに起きやすい理由**
断言を避けて中立的・丁寧に見せようとするため、主張が弱まりやすい。

**修正ルール**
- 事実として言える範囲は断言する
- 個人意見は「私は〜と感じています」「〜だと思います」と明示して断言する
- 「かもしれません」「とも言えます」は1記事に2回以内

**修正プロンプト**
```
以下の文章に「〜かもしれません」「〜とも言えます」など、
主張を弱める表現が多く含まれています。
事実や個人の見解として言い切れる部分は断言する形にリライトしてください。

[文章をここに貼る]
```

## 7. 冒頭フックの改善

**問題**
記事の冒頭が「本記事では〜について説明します」のような宣言文で始まり、読み進める動機が生まれない。

**AIに起きやすい理由**
構造的に「目的の宣言→本文」という流れを生成しやすいため、冒頭が事務的になりやすい。

**修正ルール**
- 冒頭は「読者が共感できる状況・悩み」か「結論の先出し」で始める
- 「この記事では〜を説明します」は2文目以降に移動する
- 1〜2文で「続きを読みたくなる」状態を作る

**修正プロンプト**
```
以下の記事冒頭が「〜について説明します」という宣言文で始まっており、
読み進める動機が弱い状態です。
読者が共感できる状況・悩みの描写か、結論の先出しで始まるようにリライトしてください。

[冒頭文をここに貼る]
```

## 8. 行動を促す締めの追加

**問題**
記事がまとめで終わっていて、読者が「次に何をすればいいか」分からない。

**AIに起きやすい理由**
「まとめ」として内容を整理することに集中し、読後の行動設計が抜け落ちやすい。

**修正ルール**
- 締めには「次のステップ」「試してみてほしいこと」を1つ具体的に示す
- 「ぜひ試してみてください」だけで終わらず、何を・どうするかを書く
- 関連記事へのリンクも次のアクションとして活用する

**修正プロンプト**
```
以下の記事の締めが「まとめ」だけで終わっています。
読者が記事を読んだ後に「次に何をすればいいか」が分かるよう、
具体的な次のアクション（操作・確認・関連記事など）を1つ加えてリライトしてください。

[締め部分をここに貼る]
```

## 9. 文章リズムの調整

**問題**
文の長さが均一で、読んでいて単調・つかれる。または短文が続きすぎてぶつ切り感がある。

**AIに起きやすい理由**
一定の文長を保つ傾向があり、長短のメリハリが出にくい。

**修正ルール**
- 長文（40〜60字）と短文（20字以下）を意識的に混ぜる
- 説明→結論 の流れでは、結論文を短くする
- 3文以上短文が続く場合は、1文を少し長くしてつなぎを作る

**修正プロンプト**
```
以下の文章は文の長さが均一で、読んでいてリズムが単調です。
長い文と短い文を意識的に交互に配置し、
読みやすいリズムになるようにリライトしてください。

```
